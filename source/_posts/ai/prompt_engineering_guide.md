# 提示词工程与上下文工程

> 给 AI 的大脑装上"缰绳"——从不会问到问得准，从答非所问到精准输出

***

## 前置知识

你已经学完了 LLM 的基础认知和 Ollama 本地调用，知道了：
- LLM 的本质是"文字接龙"——根据上下文猜下一个词
- 上下文窗口有限，模型不记得之前的对话
- 温度参数控制"创造力"，幻觉无法完全消除

现在的问题来了：**既然模型只是一个"文字接龙高手"，为什么有人说它能写代码、能分析数据、能做策划？它不是只会猜字吗？**

答案就是——**你喂给它什么上下文，它就在那个上下文里变得"多聪明"**。

***

## 一、从一个失败的 Prompt 开始

> **实验**：在你的 Ollama 里运行 `ollama run qwen2.5:7b`，然后输入：

```
帮我写周报
```

你会发现 AI 的回答大概率是这样的：

> 好的，以下是我的周报：
> - 本周完成了各项工作任务
> - 学习了新的技能
> - 下周计划继续努力……

这跟没写一样。不是你笨，而是**你的 Prompt 没有给 AI 任何有用的上下文**。

现在换一种问法：

```
你是一个后端开发工程师。本周完成了以下工作：
1. 用户登录模块重构，用 JWT 替换了原来的 session，性能提升 30%
2. 修复了支付接口偶发超卖的 bug
3. 参与了需求评审，讨论了 V2 版本的权限设计

下周计划：开发支付接口退款功能
当前问题：第三方支付文档更新不及时，对接进度受阻

要求：
- 300 字左右
- 语气专业但不刻板
- 分"本周完成"、"下周计划"、"问题与风险"三段
```

这次 AI 会产出一份可以直接发给领导的周报。

**思考**：两个 Prompt 问的是同一件事，为什么结果天差地别？

> 因为 LLM 没有"意图"——它不知道你是谁、你要给谁看、你想要什么格式。你给它的上下文越多，它在这个上下文里"扮演"得就越像一个真正的人。

***

## 二、什么是提示词工程（Prompt Engineering）📖

### 通俗理解

提示词工程就是**研究怎么提问才能让 AI 给出最好的回答**。

**类比**：AI 就像一个能力超强的实习生，但它没有常识。你跟它说"帮我做个报告"，它不知道你要什么；你说"帮我做个 PPT，面向投资人，讲 Q3 销售数据，10 页以内，风格商务简洁"，它就能做出一份靠谱的 PPT。

**关键认知**：Prompt Engineering 不是"魔法咒语"，而是**信息传达的艺术**——把你脑子里的需求，翻译成模型能理解的指令。

### 一个好 Prompt 的四要素

你在 LLM 入门中已经见过这个公式了，这里深入拆解：

| 要素 | 作用 | 缺失时会怎样 |
|------|------|-------------|
| **角色** | 给 AI 一个身份 | 回答风格飘忽不定 |
| **任务** | 告诉 AI 做什么 | 答非所问 |
| **背景信息** | 提供上下文 | 回答太泛、太浅 |
| **输出要求** | 规定格式和风格 | 格式不对、长篇大论 |

### 逐步拆解：四要素缺一不可

来看一个反面教材：

```
写一个 Python 函数
```

**问题 1**：谁在说话？（角色）→ 不知道，可能是程序员、可能是学生、可能是产品经理。AI 会用最通用的语气回答。

**问题 2**：写什么函数？（任务）→ 不知道，函数可以干任何事。

**问题 3**：为什么要写？（背景）→ 不知道，可能是在做作业，可能是在做生产系统。

**问题 4**：写成什么样？（输出）→ 不知道，可能 3 行也可能 300 行，可能带注释也可能不带。

现在加上四要素：

```
你是一个有 5 年经验的 Python 后端工程师（角色）。
请写一个 FastAPI 路由函数，处理用户注册请求（任务）。
用户提交 JSON 包含 username、email、password 三个字段。
需要校验邮箱格式、密码长度至少 8 位、用户名不能重复（假设数据库 User 表已存在）。
注册成功后返回 201 和用户 ID，失败返回 400 和错误信息（背景+输出）。
要求：
- 使用 Pydantic 做数据校验
- 加上类型注解和简短注释
- 不要写完整的 main.py，只写这个路由函数（输出要求）
```

**思考**：对比两个 Prompt，第二个为什么好？

> 它把一个模糊的请求变成了一个精确的"任务书"。AI 不需要猜你的意图，因为意图已经写明白了。

## 三、Prompt 进阶技巧

### 3.1 Zero-shot vs Few-shot

**Zero-shot（零样本）**：直接问，不给例子。

```
把这段话翻译成英文："今天天气真好"
```

大多数情况下够用了，但遇到复杂任务时，AI 可能不理解你要的格式。

**Few-shot（少样本）**：给几个例子，让 AI "照葫芦画瓢"。

```
你是一个文本分类助手。请将用户评论分为"好评"、"中评"、"差评"三类。

示例：
"东西很好，物流也快" → 好评
"一般般吧，不值这个价" → 中评
"质量太差了，用了一天就坏了，退货！" → 差评

请分类：
"客服态度很差，发消息半天不回"
```

**思考**：为什么给几个例子效果会更好？

> 因为 Few-shot 利用了 LLM 的"模式识别"能力。模型看到三个"输入→输出"的例子后，自动总结出分类的规律——它不是在理解规则，而是在模仿你给出的模式。这就是 LLM "in-context learning（上下文学习）"的核心机制。

### 3.2 消息的三种角色（Message Roles）

你在调用 OpenAI SDK、LangChain、Dify 或 Ollama 时，会发现消息不是一条简单的字符串，而是带有 `role` 字段的结构体：

```python
messages = [
    {"role": "system",    "content": "你是一个专业的律师"},
    {"role": "user",      "content": "劳动合同试用期最长多久？"},
    {"role": "assistant", "content": "根据《劳动合同法》第十九条，试用期最长不超过 6 个月。"},
    {"role": "user",      "content": "那 3 年合同呢？"},
]
```

这三种 `role` 分别代表对话中的不同身份。理解它们是写好 Prompt 的基础：

| 角色 | 常见名称 | 谁在说话 | 作用 | 类比 |
|------|---------|---------|------|------|
| **system** | System Prompt / Developer Message | 开发者（你） | 设定 AI 的人设、行为边界、全局规则 | 员工手册 |
| **user** | Human Message / User Prompt | 用户 | 提出问题或给出指令 | 老板布置的任务 |
| **assistant** | AI Message / Assistant Message | AI 模型 | 模型之前的回答（用于多轮对话） | 员工完成的工作记录 |

**System Message（系统消息）**：
- 通常在对话开始时设置，**只出现一次**
- 影响整个对话的基调，模型会把它的优先级视为最高
- 不同平台叫法不同：OpenAI 叫 `system`，Anthropic 叫 `developer`，Dify 里叫"系统提示词"

**User / Human Message（用户消息）**：
- 就是用户说的话，可以出现多次
- 多轮对话中，每一轮用户的输入都会作为一条 `user` 消息加入
- OpenAI SDK 叫 `user`，LangChain 里叫 `HumanMessage`，本质完全一样

**Assistant / AI Message（AI 消息）**：
- 这是 AI 自己之前生成的回答
- **为什么要传回去？** 因为模型不记得之前的对话，你需要把它的历史回答作为上下文传回来，它才能理解"你在接着聊什么"
- 多轮对话的实现原理就是：`[system, user₁, assistant₁, user₂, assistant₂, ..., userₙ]` → 生成 `assistantₙ`

**一个完整的多轮对话示例**：

```python
import ollama

messages = [
    # system：设定人设（整个对话只出现一次）
    {"role": "system", "content": "你是一个耐心的数学老师，用通俗易懂的方式解释概念。"},
    
    # 第一轮对话
    {"role": "user", "content": "什么是勾股定理？"},
    {"role": "assistant", "content": "勾股定理说的是：直角三角形中，两条直角边的平方和等于斜边的平方..."},
    
    # 第二轮对话
    {"role": "user", "content": "那如果一个直角边是 3，另一个是 4，斜边是多少？"},
    # 下一轮 AI 的回答会基于上面所有消息来生成
]

response = ollama.chat(model="qwen2.5:7b", messages=messages)
```

> **初学者常犯的错**：只传 `user` 消息，不传 `assistant` 历史回答。结果就是 AI 完全不记得之前聊过什么——因为从它的视角看，每次调用都是第一次见面。

### 3.3 系统提示词（System Prompt）的进阶用法

System Prompt 的作用前面已经说过了，这里补充几个实际开发中的高级用法：

**技巧 1：用 System Prompt 约束输出格式**

```python
{"role": "system", "content": "你只输出 JSON 格式，不要任何额外文字。JSON 结构包含 name、age、hobbies 三个字段。"}
```

**技巧 2：用 System Prompt 设定回复风格**

```python
{"role": "system", "content": "用小学生能听懂的语言解释，不要用专业术语。如果必须用专业术语，请在括号中用大白话解释。"}
```

**技巧 3：不同平台的 System Prompt 差异**

| 平台 | 字段名 | 特殊说明 |
|------|-------|---------|
| OpenAI | `system` | 最标准的用法 |
| Anthropic Claude | `system`（API 参数，不在 messages 里） | Claude 把 system 单独抽成一个 API 参数 |
| Ollama | `system`（在 messages 里） | 与 OpenAI 格式一致 |
| LangChain | `SystemMessage` | Python 类，底层转为 `role: "system"` |
| Dify | 设置面板中的"系统提示词" | 不需要写代码，在 UI 中填写即可 |

> 不管叫什么名字、放在哪里，核心概念只有一个：**System Prompt 是你对 AI 说的第一句话，它决定了整场对话的基调。**

### 3.4 约束与边界

一个好的 Prompt 不仅要告诉 AI "做什么"，还要告诉它 "不要做什么"：

```
请总结以下文章的核心观点。
要求：
- 不超过 200 字
- 只基于原文，不要添加外部知识
- 如果你不确定某个观点，请标注"存疑"
- 不要使用"首先、其次、最后"等套话
- 如果文章内容涉及非法或不道德的行为，请拒绝总结并说明原因
```

**关键技巧**：
- `"不要 X"` 比 `"请做 Y"` 更容易被模型忽略，所以优先用正向描述
- 在 Prompt 末尾重复关键约束（模型对末尾内容注意力更高）
- 让模型"先思考再回答"（见下文思维链）

### 3.5 提示词工程 vs 上下文工程：一张表看懂区别

很多初学者分不清这两个概念，因为它们确实有重叠。但它们的**关注点和粒度**完全不同：

| 维度 | 提示词工程（Prompt Engineering） | 上下文工程（Context Engineering） |
|------|------|------|
| **关注什么** | 怎么问 | 喂什么 |
| **粒度** | 单次交互（一条对话） | 多轮交互（整个会话生命周期） |
| **解决的核心问题** | "怎么表达，模型才懂我的意图？" | "窗口有限，什么该放进来、什么该扔掉？" |
| **操作对象** | 用户写的这一条消息 | 系统提示 + 历史记录 + 知识库 + 工具描述 + 当前消息 |
| **时间维度** | 静态（写一次，发一次） | 动态（随着对话推进不断更新） |
| **类比** | 给实习生写一封清晰的工作邮件 | 管理整个项目的档案柜——哪些文件放桌面、哪些归档、哪些销毁 |

**思考**：回到开头写周报的例子，哪些属于提示词工程、哪些属于上下文工程？

```
你是一个后端开发工程师。本周完成了以下工作：
1. 用户登录模块重构...（上下文工程：注入了工作记录数据）
2. ...
要求：300 字左右，分三段。（提示词工程：设计提问方式）
```

> 在实际应用中，两者**不可分割**：提示词工程决定了"问得好不好"，上下文工程决定了"答得准不准"。一个好的 AI 应用必须两者兼顾。

## 四、什么是上下文工程（Context Engineering）📖

### 4.1 问题的根源：上下文窗口是有限的

你已经知道上下文窗口（Context Window）是模型一次能处理的最大 Token 数。即便是 128K 的窗口，也不可能无限塞内容。

**现实问题**：
- 一份 500 页的合同 ≈ 30 万 Token，远超所有模型的窗口
- 一个项目有几万行代码，不可能一次全塞进 Prompt
- 长对话到 50 轮之后，模型开始"忘"前面的内容

**上下文工程就是研究：在有限的窗口里，怎么安排最有价值的信息，让模型发挥最大能力。**

**类比**：上下文窗口就像你的"工作记忆"——你同时能记住的东西有限。上下文工程就是决定"现在应该想起什么、忘掉什么"。

### 4.2 上下文工程的三大场景

#### 场景一：文档太长怎么办？

你有一份 200 页的产品手册，想让 AI 帮你回答用户问题。

**错误做法**：把整个手册塞进 Prompt → 窗口溢出，或者模型注意力分散。

**正确做法（RAG 的前置思想）**：
1. 把手册切成一段一段的（每段 500-1000 字）
2. 用户提问时，先找出最相关的几段
3. 只把这几段作为上下文喂给模型
4. 让模型基于这几段回答

> 这就是之后要深入讲的 **RAG（检索增强生成）** 📖。上下文工程在这里的决策是：**不喂全量文档，只喂与当前问题最相关的片段。**

#### 场景二：长对话怎么维护？

你的 AI 客服跟用户聊了 100 轮，第 101 轮时模型已经开始"忘"第 1 轮说的话了。

**策略 1：滑动窗口**
只保留最近 N 轮对话（比如最近 10 轮），超过的直接丢弃。
- **优点**：简单
- **缺点**：早期的关键信息（如用户说"我要退货"）被丢弃

**策略 2：摘要压缩**
每隔一定轮数，让 AI 把之前的对话压缩成摘要：

```
原始对话（20 轮）→ AI 自动总结为：
"用户张三，订单号#12345，购买 iPhone 16，反映屏幕有坏点，要求退货，已确认在退换期内。"
```

然后把摘要 + 最近几轮对话一起作为上下文传入。

**策略 3：关键信息提取**
在对话过程中主动抽取关键信息存储：

```python
# 每次对话后提取关键信息
key_facts = extract_facts(conversation_history)
# 存储到数据库或向量库中
save_to_memory(user_id, key_facts)
# 下次对话时从记忆中取回
context = load_from_memory(user_id) + recent_messages
```

**思考**：如果让你设计一个 AI 客服的上下文管理系统，你会选哪种策略？为什么？

> 没有标准答案，实际工程中最常见的是**策略 1 + 策略 3 组合**：保留最近 N 轮对话 + 从存储中召回关键事实。策略 2（摘要压缩）适合对上下文窗口极小的场景。

#### 场景三：多工具调用时怎么选上下文？

你的 Agent 有 20 个可用工具（查天气、发邮件、查数据库、调 API……），但上下文窗口有限，不能把所有工具的说明都塞进 Prompt。

**策略**：
- **动态工具注入**：根据用户意图，只把相关工具的说明加入上下文
- **工具分类索引**：先把工具分成几类，每类用一句话描述，用户选择具体类别后再注入详细工具说明

> 这就是 **Harness Engineering（马具工程）** 📖 的范畴——标准化地管理 AI 的输入、输出、工具调用和错误处理。

### 4.3 上下文工程的注意力规律

LLM 对上下文不同位置的关注度不一样：

```
Prompt 开头 ←←← 中间 ←←← 结尾
注意力：  高          低          高
```

这是已被研究证实的"中间遗忘"现象——模型对开头和结尾的内容注意力最高，中间部分容易忽略。

**实操建议**：
1. **最重要的约束放开头**（角色、核心任务）
2. **最关键的格式要求放末尾**（输出格式、字数限制）
3. **避免把核心指令埋在中间**

**验证实验**：同一个 Prompt，把"不要使用套话"放在开头 vs 放在中间，观察 AI 是否真的遵守。你会发现放在末尾的效果最好。

## 五、实战：Prompt 迭代优化

Prompt 不是一次写成的，而是**通过反复测试逐步优化**的。以下是一个真实的迭代过程：

### 需求：让 AI 根据用户反馈生成产品改进建议

**v1（太模糊）**：
```
根据用户反馈生成产品改进建议
```

**v2（加了角色和输出要求）**：
```
你是一个产品经理。请根据以下用户反馈，生成 5 条产品改进建议。
按优先级从高到低排序。

用户反馈：
[粘贴反馈内容]
```

**v3（加了背景信息和约束）**：
```
你是一个有 3 年经验的 B 端 SaaS 产品经理。
请分析以下用户反馈，按以下格式输出：

1. 问题归类（功能缺陷 / 体验优化 / 新功能需求）
2. 受影响用户比例（根据反馈内容估算）
3. 改进建议（具体可执行的方案，200 字以内）
4. 优先级（P0 紧急 / P1 高 / P2 中 / P3 低）

用户反馈：
[粘贴反馈内容]

注意：
- 改进建议必须具体可落地，不要"提升用户体验"这种空话
- 如果是明显不合理的需求（如"免费使用所有高级功能"），标注为"不合理"并说明原因
```

**v4（加了 Few-shot 示例）**：
```
你是一个有 3 年经验的 B 端 SaaS 产品经理。
请分析以下用户反馈，按以下格式输出：

格式示例：
---
反馈："报表导出太慢了，每次都要等 5 分钟"
分析：
- 问题归类：功能缺陷
- 受影响用户比例：约 40%（财务和销售岗位高频使用导出）
- 改进建议：引入异步导出机制——用户点击导出后立即返回"导出任务已提交"，完成后通过站内消息通知用户下载。可将等待时间从 5 分钟降至 10 秒。
- 优先级：P1 高
---

请分析：
反馈：[新的反馈内容]

注意：
- 改进建议必须具体可落地，不要"提升用户体验"这种空话
- 如果是明显不合理的需求，标注为"不合理"并说明原因
```

**思考**：从 v1 到 v4，每一步加了什么？为什么效果更好？

> v2 加了角色和格式 → AI 知道"我是谁、输出长什么样"
> v3 加了具体约束和排错逻辑 → AI 知道"什么不能说、遇到边界怎么办"
> v4 加了示例 → AI 通过模仿掌握了分析的"颗粒度"和"语气"

### 6.1 Prompt 模板化

在实际开发中，不要硬编码 Prompt，而是使用模板：

```python
PROMPT_TEMPLATE = """
你是一个{role}。

背景信息：
{context}

输出要求：
- {output_requirement}
"""

def generate_prompt(role, task, context, output_requirement):
    return PROMPT_TEMPLATE.format(
        role=role,
        task=task,
        context=context,
        output_requirement=output_requirement

```

### 6.2 上下文拼接策略

```python
def build_context(system_prompt, history, knowledge, user_input):
    """
    构建完整的上下文，按注意力权重排列：
    1. 系统提示（最高优先级）
    2. 检索到的知识片段
    3. 最近 N 轮历史对话
    4. 当前用户输入（末尾最高注意力）
    """
    context_parts = []
    
    # 系统提示放开头
    context_parts.append(system_prompt)
    
    # 知识片段（RAG 检索结果）
    if knowledge:
        context_parts.append(f"## 参考资料\n{knowledge}")
    
    # 历史对话（最近 N 轮）
    if history:
        context_parts.append(f"## 历史对话\n{history}")
    
    # 用户输入放末尾（最高注意力位置）
    context_parts.append(f"## 当前问题\n{user_input}")
    
    return "\n\n".join(context_parts)
```

### 6.3 Token 计数与截断

```python
import tiktoken  # OpenAI 的 Token 计数器

def truncate_to_fit(texts, max_tokens, model="cl100k_base"):
    """确保总 Token 数不超过限制，超出时从历史消息开始截断"""
    encoder = tiktoken.get_encoding(model)
    total = sum(len(encoder.encode(t)) for t in texts)
    
    if total <= max_tokens:
        return texts
    
    # 从倒数第二个开始截断（保留最后一个，即用户输入）
    for i in range(len(texts) - 2, 0, -1):
        texts[i] = texts[i][:len(texts[i]) // 2]
        total = sum(len(encoder.encode(t)) for t in texts)
        if total <= max_tokens:
            break
    
    return texts
```

## 七、总结

### 提示词工程一句话

> **把 AI 当成一个能力强但没有常识的实习生——你说得越清楚，它做得越好。**

### 上下文工程一句话

> **上下文窗口是有限的"工作记忆"，上下文工程就是决定往里塞什么、丢掉什么、压缩什么。**

## 附录：Prompt 速查表

### 常用指令模式

| 目的 | Prompt 模板 |
|------|-----------|
| **摘要** | "请用 {N} 字总结以下内容：[内容]" |
| **翻译** | "请将以下内容从{源语言}翻译为{目标语言}，保持{正式/口语/文学}风格：[内容]" |
| **分类** | "请将以下{文本}分为{类别}，格式为'原文→类别'：[文本]" |
| **生成** | "你是一名{角色}。请生成{内容类型}，要求{约束}：[主题/要点]" |
| **改写** | "请将以下内容改写为{目标风格}，{保留/删除}{元素}：[内容]" |
| **代码** | "用{语言}写一个{功能}，输入{参数}，输出{结果}，要求{约束}：[细节]" |
| **分析** | "请分析以下内容，从{维度1}、{维度2}两个角度给出结构化报告：[内容]" |

### 避坑清单

| ❌ 错误做法 | ✅ 正确做法 |
|-----------|-----------|
| 模糊指令："写篇文章" | 具体指令："以 XX 身份，就 XX 主题，写一篇 XX 风格的文章" |
| 不限制输出长度 | 指定"不超过 N 字 / N 个要点" |
| 所有约束放中间 | 关键约束放开头或末尾 |
| 一次给太多任务 | 拆成多个 Prompt 逐步完成 |
| 不给示例 | 复杂任务给 1-3 个 Few-shot 示例 |
| 让模型"自由发挥" | 规定格式、风格、边界条件 |
